# External Agent Integration

This page is for agents running outside this repository context (remote assistants, IDE agents, orchestration services).

## Goal

Make external agents use both:

- `skills/` for workflow and guardrails
- `mantle-mcp` for deterministic data and typed interfaces

The local `skills/` path is backed by the `mantle-xyz/mantle-skills` Git submodule, and the actual skill folders live under `skills/skills/`.

## Integration Checklist

1. Build and run MCP server:
   - `npm install`
   - `npm run build`
   - launch `node dist/index.js` with `MANTLE_MCP_TRANSPORT=stdio`
2. Register MCP server in agent runtime config.
3. Run `npm run skills:init` if the `skills/` submodule has not been initialized yet.
4. Give agent read access to repository `skills/` folder.
5. Require skill-first workflow before tool execution.
6. Require MCP outputs as final-answer evidence.

## Skill-First Protocol (Required)

For every user task:

1. Select matching skill from `skills/skills/*`.
2. Read `skills/skills/<skill-name>/SKILL.md`.
3. Apply its checklist.
4. Execute relevant MCP tools/resources/prompts.

If no exact skill matches, external agent should:

1. Choose closest skill for risk controls.
2. State assumptions.
3. Continue with MCP-grounded execution.

## MCP Execution Protocol (Required)

External agents should follow this sequence:

1. Discover:
   - `listTools`
   - `listResources`
   - `listPrompts`
2. Load context:
   - `readResource` for `mantle://...` URIs
   - `getPrompt` for workflow templates
3. Execute:
   - `callTool` with schema-valid arguments
4. Report:
   - cite tool outputs
   - do not invent addresses, balances, chain status, or token prices

## Recommended Tool Order

1. Chain context: `mantle_getChainInfo`, `mantle_getChainStatus`
2. Canonical resolution: `mantle_resolveAddress`, `mantle_resolveToken`
3. Domain actions: account/token/defi/indexer/diagnostics tools
4. Error handling: use typed error payload fields (`code`, `message`, `suggestion`, `details`)

## Minimal Safety Policy for External Agents

- Always resolve token/address before transaction-adjacent reasoning.
- Treat `null` price as unknown data.
- For custom endpoints, rely on built-in endpoint policy enforcement.
- Use diagnostics tools before declaring endpoint health.

## Python Integration with pydantic-ai

[pydantic-ai](https://ai.pydantic.dev/) connects to mantle-mcp via `MCPServerStdio`. The agent discovers all 19 tools automatically through the MCP protocol — no manual tool definitions needed.

### Minimal Example

```python
from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio

mantle_server = MCPServerStdio(
    "node",
    args=["path/to/mantle-agent-scaffold/dist/src/index.js"],
    env={
        "MANTLE_MCP_TRANSPORT": "stdio",
        "MANTLE_RPC_URL": "https://rpc.mantle.xyz",
    },
    timeout=30,
)

agent = Agent(
    "anthropic:claude-sonnet-4-20250514",
    toolsets=[mantle_server],
    instructions="You are a Mantle L2 assistant. Gas token is MNT, not ETH.",
)


async def main():
    async with agent:
        result = await agent.run("What is the current gas price on Mantle?")
    print(result.output)
```

The `async with agent` context manager starts the MCP server subprocess, calls `listTools` to discover available tools, and shuts it down on exit. Inside the context, `agent.run()` can invoke any of the 19 mantle-mcp tools as needed.

### Skill-First Integration

For production agents, load skill content into the system prompt so the LLM follows Mantle-specific workflows and guardrails:

```python
from pathlib import Path
from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio

SKILLS_DIR = Path("path/to/mantle-agent-scaffold/skills/skills")

mantle_server = MCPServerStdio(
    "node",
    args=["path/to/mantle-agent-scaffold/dist/src/index.js"],
    env={"MANTLE_MCP_TRANSPORT": "stdio"},
    timeout=30,
)


def load_skill(name: str) -> str:
    """Load a skill's SKILL.md content."""
    skill_path = SKILLS_DIR / name / "SKILL.md"
    return skill_path.read_text()


agent = Agent(
    "anthropic:claude-sonnet-4-20250514",
    toolsets=[mantle_server],
    instructions=(
        "You are a Mantle L2 DeFi assistant. "
        "Always follow the skill workflow below.\n\n"
        + load_skill("mantle-portfolio-analyst")
    ),
)


async def main():
    async with agent:
        result = await agent.run(
            "Analyze the portfolio for 0x742d35Cc6634C0532925a3b844Bc9e7595f2bD18"
        )
    print(result.output)
```

### Dynamic Skill Selection

For agents that handle multiple task types, select the skill at runtime based on user intent:

```python
SKILL_ROUTING = {
    "portfolio": "mantle-portfolio-analyst",
    "swap": "mantle-defi-operator",
    "risk": "mantle-risk-evaluator",
    "debug": "mantle-readonly-debugger",
    "network": "mantle-network-primer",
    "address": "mantle-address-registry-navigator",
}


def select_skill(user_message: str) -> str | None:
    """Match user intent to the narrowest applicable skill."""
    msg = user_message.lower()
    for keyword, skill_name in SKILL_ROUTING.items():
        if keyword in msg:
            return skill_name
    return None
```

This follows the skill-first protocol: select the narrowest matching skill, load its checklist, then execute MCP tools with the skill's guardrails applied.

### Multiple MCP Servers

pydantic-ai supports multiple MCP servers in a single agent. Combine mantle-mcp with other tool servers:

```python
from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio

mantle = MCPServerStdio(
    "node",
    args=["path/to/mantle-agent-scaffold/dist/src/index.js"],
    env={"MANTLE_MCP_TRANSPORT": "stdio"},
    timeout=30,
)

# Add other MCP servers as needed
tavily = MCPServerStdio("npx", args=["-y", "tavily-mcp"], timeout=30)

agent = Agent(
    "anthropic:claude-sonnet-4-20250514",
    toolsets=[mantle, tavily],
)
```

### Production Pattern: Multi-Agent Orchestrator

In production systems, agents often use an orchestrator that delegates to specialist sub-agents. Each specialist can have its own MCP servers configured via YAML manifests. This pattern was validated in a real Telegram bot integration:

```python
# manifests/research.yaml — declarative MCP server config per agent
"""
name: research
model:
  primary: openai:gpt-4o
tools:
  - web_search
  - crypto_price
mcp_servers:
  - tavily
  - mantle          # <-- declares which MCP servers this agent needs
"""
```

The agent factory reads the manifest and connects MCP toolsets automatically:

```python
from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio


# MCP server registry — one entry per server
MCP_SERVERS = {
    "mantle": MCPServerStdio(
        "node",
        args=["/app/mantle-mcp/dist/src/index.js"],
        env={
            "MANTLE_MCP_TRANSPORT": "stdio",
            "MANTLE_RPC_URL": "https://rpc.mantle.xyz",
        },
        timeout=30,
    ),
    "tavily": MCPServerStdio("npx", args=["-y", "tavily-mcp"], timeout=30),
}


def build_agent(manifest) -> Agent:
    """Build agent with MCP toolsets from manifest declaration."""
    # Resolve MCP servers declared in the manifest
    toolsets = [MCP_SERVERS[name] for name in manifest.mcp_servers if name in MCP_SERVERS]

    agent = Agent(
        model=manifest.model,
        instructions=manifest.system_prompt,
        toolsets=toolsets or None,
    )

    # Register non-MCP tools from a tool registry
    for tool_fn in get_tools(manifest.tools):
        agent.tool(tool_fn)

    return agent
```

#### Contextual Instructions

LLMs may ignore MCP tools if they don't know when to use them. Inject context-specific instructions when the user message matches relevant keywords:

```python
MANTLE_KEYWORDS = ["mantle", "mnt", "agni", "aave mantle", "defi mantle"]

MANTLE_INSTRUCTIONS = """Mantle L2 (via MCP tools — 19 instruments, real on-chain data):
- Gas token = MNT, not ETH.
- Use mantle_resolveAddress/mantle_resolveToken to verify addresses. Never guess.
- Use mantle_getTokenPrices for token prices. Don't rely on memory.
- Portfolio: mantle_getBalance (native) + mantle_getTokenBalances (ERC-20).
- DeFi: mantle_getSwapQuote, mantle_getPoolLiquidity, mantle_getLendingMarkets.
- All tools are read-only — analyze and recommend, never execute transactions."""


@coordinator.system_prompt
async def inject_mantle_context(ctx):
    """Inject Mantle instructions only when the message is Mantle-related."""
    message = (ctx.deps.current_message or "").lower()
    if any(kw in message for kw in MANTLE_KEYWORDS):
        return MANTLE_INSTRUCTIONS
    return ""
```

#### Docker Deployment

Mount the pre-built `dist/` directory as a read-only volume — no need to install Node.js dependencies at runtime:

```yaml
# docker-compose.yml
services:
  bot:
    build: .
    volumes:
      - ./mantle-mcp-dist:/app/mantle-mcp:ro  # read-only mount
    environment:
      - MANTLE_MCP_PATH=/app/mantle-mcp/src/index.js
```

#### Security Checklist

- **Minimal subprocess env**: only pass `PATH`, `HOME`, `MANTLE_MCP_TRANSPORT`, `MANTLE_RPC_URL` to the MCP subprocess — never leak API keys or database credentials
- **Path validation**: verify `MANTLE_MCP_PATH` stays within the expected directory before spawning the subprocess
- **Read-only mount**: use `:ro` in Docker to prevent code modification from inside the container
- **Graceful degradation**: if the MCP server fails to start, the agent should continue working without Mantle tools

#### Lessons Learned

1. **MCP toolsets must be explicitly connected** — declaring `mcp_servers` in a manifest is not enough. The agent factory must pass them as `toolsets=` to the `Agent()` constructor.
2. **LLMs need prompting to use MCP tools** — without contextual instructions mentioning `mantle_*` tool names, the LLM will default to familiar tools like web search even when MCP tools would give better results.
3. **First MCP call is slow** — the MCP subprocess starts on first use (not at agent creation). Budget 5–15 seconds for the initial `listTools` handshake in timeout settings.

### Key Notes

- **Lifecycle**: `async with agent` manages MCP subprocess start/stop. Without it, tools are not available.
- **Tool discovery**: all 19 mantle-mcp tools are registered automatically via the MCP `listTools` handshake.
- **Error handling**: tool errors return structured payloads with `code`, `message`, `suggestion`, and `details` fields.
- **No private keys**: mantle-mcp is read-only. Transaction tools return unsigned payloads only.
